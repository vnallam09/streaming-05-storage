## Phase 4: Technical Modification

I started from Dr. Case's case consumer
(`src/streaming/kafka_consumer_case.py`) and made one small, focused change.

- **New file:** `src/streaming/kafka_consumer_teja.py`
- **The change:** store an **additional field in DuckDB** - the derived order
  **`total`**.

In Dr. Case's case example, the consumer computes the derived `total` for every
sale but only writes it to the output CSV. The DuckDB valid table stores the raw
and optional fields plus Kafka metadata, but **not** `total`. My copy adds
`total` as a stored column so it can be queried directly with SQL.

To keep Dr. Case's example untouched, the storage change is self-contained
inside `kafka_consumer_teja.py`:

- a new table `consumed_valid_sales_teja`
- field list `TEJA_VALID_FIELDNAMES` = the case valid fields **+ `total`**
- helpers `init_db_teja()`, `write_valid_record_teja()`, and
  `log_storage_summary_teja()`

Because `total` is now a column, the consumer's end-of-run summary can ask the
database a question the case version could not answer from DuckDB alone:

```sql
SELECT region_id,
       COUNT(*) AS sale_count,
       ROUND(SUM(CAST(total AS DOUBLE)), 2) AS total_sales
FROM consumed_valid_sales_teja
GROUP BY region_id
ORDER BY total_sales DESC;
```

Run it with:

```shell
uv run python -m streaming.kafka_consumer_teja
```

## Phase 5: Apply the Skills to a New Problem

For Phase 5 I applied the same streaming-storage pattern (produce → validate →
enrich → store in DuckDB) to a **completely different dataset**, so nothing in
Dr. Case's sales example or my Phase 4 file is affected.

### Scenario / dataset chosen

**Wearable fitness workouts.** A stream of workout sessions from a fitness
watch, enriched with per-athlete reference data.

| File | Purpose |
| --- | --- |
| `data/workouts.csv` | Event stream - one workout session per row |
| `data/workout_users.csv` | Reference table - athlete profile (weight, max HR) |
| `src/streaming/data_validation/data_contract_workouts.py` | Workouts data contract |
| `src/streaming/data_engineering/workout_fields.py` | Derived field calculations |
| `src/streaming/storage/storage_workouts.py` | DuckDB storage for workouts |
| `src/streaming/kafka_producer_workouts_teja.py` | Produces workout messages |
| `src/streaming/kafka_consumer_workouts_teja.py` | Consumes, validates, enriches, stores |

### What changed from the case example

- **New domain and topic.** Messages flow on the dedicated topic
  `streaming-05-storage-workouts`, so Dr. Case's `streaming-05-storage-case`
  topic is never touched. The producer and consumer override the topic with
  `dataclasses.replace(KafkaSettings.from_env(), topic=...)`.
- **New reference data.** Instead of regions → tax rate, the lookup is
  user_id → `{weight_kg, max_hr}`. This is a richer lookup (two values per key)
  used for enrichment.
- **New derived fields** (computed by the consumer, not sent by the producer):
  - `calories` - MET formula: `MET × weight_kg × hours`
  - `pace_min_per_km` - minutes per km (0 when distance is 0)
  - `intensity_pct` - average HR as a percent of the athlete's max HR
  - `hr_zone` - training zone 1-5 derived from `intensity_pct`
- **New DuckDB tables** - `consumed_valid_workouts` and
  `consumed_rejected_workouts`, in their own database file
  `data/output/workouts.duckdb`.

### What the consumer does to each message

1. **Validates** it against the workouts data contract: required fields present,
   `user_id` known, `activity` in the allowed set, valid ISO `datetime`,
   positive `duration_min` and `avg_hr`, and a non-negative `distance_km`.
2. **Enriches** valid messages with the four derived fields above, using the
   athlete profile lookup.
3. **Stores** valid records (raw + derived + Kafka metadata) in
   `consumed_valid_workouts`, and **also stores rejected records** - with their
   validation errors - in `consumed_rejected_workouts` for later analysis.
4. **Writes** each valid record to `data/output/consumed_workouts.csv`.

### What fields I stored

The valid table stores every raw/optional field **plus the derived fields**
`calories`, `pace_min_per_km`, `intensity_pct`, and `hr_zone`, plus the Kafka
metadata (`_kafka_key`, `_kafka_partition`, `_kafka_offset`). Storing the
derived fields is what makes the stored stream useful for SQL analysis.

### What I learned from querying the stored stream

Running the pipeline over the sample data accepted **14** sessions and rejected
**4** (an unknown athlete, an unsupported `yoga` activity, an aborted
zero-duration session, and a non-numeric duration) - exactly the kinds of bad
records a data contract is meant to catch.

Because the derived fields are stored, the consumer's summary answers real
questions with SQL. **Effort by activity:**

| activity | sessions | minutes | total kcal | avg intensity |
| --- | --- | --- | --- | --- |
| cycle | 4 | 265 | 2,446.6 | 78.5% |
| run | 5 | 203 | 2,263.2 | 86.8% |
| swim | 2 | 85 | 770.3 | 78.5% |
| walk | 2 | 120 | 540.1 | 58.8% |
| row | 1 | 30 | 278.2 | 76.8% |

A nice insight: **running burns the most calories per minute** (highest average
intensity at 86.8%), while **cycling racks up the highest total** simply because
those sessions are longer. The HR-zone breakdown of the accepted sessions
(zone 1: 1, zone 2: 1, zone 3: 5, zone 4: 7) shows most training sat in the
moderate-to-hard zones 3-4 - the story you only get once the derived fields are
**stored**, not just logged and discarded.

Run it with (producer first in one terminal, consumer in another):

```shell
uv run python -m streaming.kafka_producer_workouts_teja
uv run python -m streaming.kafka_consumer_workouts_teja
```
